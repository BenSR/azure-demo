## 4. Detailed Gap Findings

### Gap 1: Application Insights Standard Web Test (FR-7.1 — Resolved)

**Requirement:** Implement basic health monitoring for the API (e.g., availability test or health-check endpoint).

**Previous state:** The health endpoint (`GET /api/health`) was fully implemented and functional. An `azurerm_monitor_metric_alert` for `availabilityResults/availabilityPercentage` existed in `phase3/env/alerts.tf`. However, no `azurerm_application_insights_standard_web_test` resource was deployed to generate synthetic availability data.

**Resolution:** Added `azurerm_application_insights_standard_web_test.health` resource in `phase3/env/alerts.tf` (one per stamp) targeting the health endpoint via the APIM gateway URL. The web test probes from 5 geo-locations (UK South, West Europe, South Central US, North Central US, Southeast Asia) and validates a 200 response containing "healthy".

Because APIM is in Internal VNet mode with no public endpoint, the web test is **disabled by default** (`var.web_test_enabled = false`). This is a deliberate design decision: external probes cannot reach the private gateway, so enabling the test would produce false-negative availability results. The infrastructure is fully wired — enabling requires setting `web_test_enabled = true` once a public ingress path (e.g., Application Gateway) is deployed. Supporting variables added: `web_test_frequency_seconds`, `web_test_timeout_seconds`, `web_test_geo_locations`.

### Gap 3: Infrastructure Testing (FR-10.1 / NFR-6.1 — Not Met)

**Requirement:** Include automated infrastructure tests using `terraform test` or Terratest to validate resource correctness.

**Current state:** No infrastructure test files exist in the repository. Testing is limited to:
- `terraform validate` (syntax checking only, not resource correctness)
- Python unit tests for the Function App
- End-to-end smoke test script (`Test-Application-Jumpbox.ps1`) — manual, run from the jumpbox

**Impact:** No automated verification that deployed resources match expected configurations (e.g., correct SKUs, network rules, RBAC assignments). This is a stretch goal.

### Gap 4: Application Gateway (FR-1.5 — Not Met)

**Requirement:** Deploy an Application Gateway to handle ingress traffic into the VNet.

**Current state:** Not implemented. APIM in internal VNet mode serves as the ingress layer.

**Impact:** Minimal — this is a stretch goal and the chosen APIM approach meets all core ingress requirements (internal-only access, mTLS termination, policy enforcement, load balancing). Application Gateway would add Layer-7 WAF capabilities and is appropriate for production scenarios requiring external-facing endpoints.

**Remediation:** If required, deploy an `azurerm_application_gateway` with a WAF_v2 SKU in a new dedicated subnet, with a backend pool pointing to the APIM private IP. Application Gateway would handle TLS termination and WAF inspection before forwarding to APIM.

4. **README ACR cost estimate** lists Basic tier (~£4) but Terraform deploys Premium (~£40+) — the total cost is understated.

Previously identified gaps for the App Insights web test (Gap 1) and incomplete diagnostic settings (Gap 2) have been resolved.
